# ReviewLoop

ReviewLoop v1 is a Rust CLI/daemon that automates paper submission and review retrieval for `paperreview.ai`.

## Features

- API-first submission flow for Stanford Agentic Reviewer (`paperreview.ai`)
- Token recording and automatic polling/retrieval
- Artifacts written to disk (`review.json`, `review.md`, `meta.json`)
- Git tag trigger and PDF-change trigger
- Duplicate submission guard by `backend + pdf_hash`
- Optional IMAP token watcher + manual token import
- Node + Playwright fallback submit path when API flow fails

## Commands

```bash
reviewloop init
reviewloop daemon run
reviewloop daemon run --panel false
reviewloop submit --paper-id main [--force]
reviewloop approve --job-id <job-id>
reviewloop import-token --paper-id main --token <token> [--source email]
reviewloop status [--paper-id main] [--json]
reviewloop retry --job-id <job-id>
```

## Quick Start

```bash
cargo run -- init
# edit reviewloop.toml (email, paper path, etc.)
cargo run -- submit --paper-id main
cargo run -- daemon run
```

## Config (`reviewloop.toml`)

Generated by `reviewloop init`.

Key defaults:

- `core.max_concurrency = 2`
- `core.max_submissions_per_tick = 1` (hard cap per daemon tick, protects remote site)
- `core.review_timeout_hours = 48`
- `polling.schedule_minutes = [10, 20, 40, 60]`
- `polling.jitter_percent = 10`
- `trigger.git.tag_pattern = "review-<backend>/<paper-id>/*"`
- `trigger.git.repo_dir = "."`
- `trigger.git.auto_create_tags_on_pdf_change = false`
- `trigger.git.auto_delete_processed_tags = false`
- `trigger.pdf.auto_submit_on_change = false`
- `trigger.pdf.max_scan_papers = 10` (only first N configured papers scanned per tick)
- `logging.level = "info"`
- `logging.output = "stdout"` (`stdout` | `stderr` | `file`)
- `logging.file_path = ".reviewloop/reviewloop.log"` (used when `logging.output = "file"`)

Foreground daemon (`daemon run`) supports a terminal monitor panel by default on TTY.
If you run daemon in background/service mode, set `logging.output = "file"` to persist logs.

`providers.stanford`:

- `base_url = "https://paperreview.ai"`
- `fallback_mode = "node_playwright"`
- `fallback_script = "tools/paperreview_fallback.mjs"`
- `email` required
- `venue` optional

### IMAP token matching (default + override)

IMAP listener is an optional build feature.

- Enable IMAP support: `cargo run --features imap-listener -- daemon run`
- Default build (without this feature) skips IMAP polling and avoids pulling IMAP parser dependencies.

`imap.backend_patterns` maps backend name -> regex. The first capture group is treated as token.

Default includes Stanford:

```toml
[imap.backend_patterns]
stanford = "https?://paperreview\\.ai/review\\?token=([A-Za-z0-9_-]+)"
```

You can override or add patterns:

```toml
[imap.backend_patterns]
stanford = "token=([A-Za-z0-9_-]+)"
mybackend = "review-token:\\s*([A-Za-z0-9_-]+)"
```

When a token is matched for backend `X`, daemon attaches it only to latest open job for backend `X` that has no token.

## Trigger Rules

### Git Tag Trigger

Supported formats:

- `review-<backend>/<paper-id>/<anything>` (canonical)
- `review-<backend>/<anything>` (shorthand; uses first configured paper for that backend)

Example:

- `review-stanford/main/v1`

Optional automation:

- If `trigger.git.auto_create_tags_on_pdf_change = true`, PDF-change trigger auto-creates local tags like `review-stanford/main/auto-<timestamp_ms>` before enqueuing jobs.
- If `trigger.git.auto_delete_processed_tags = true`, tags are deleted locally after they are consumed by git trigger.

### PDF Change Trigger

- Computes SHA256 of configured paper PDFs
- New hash enqueues a job
- Default status: `PENDING_APPROVAL` (manual `approve` required)

## Artifacts

For each completed job:

- `.reviewloop/artifacts/<job-id>/review.json`
- `.reviewloop/artifacts/<job-id>/review.md`
- `.reviewloop/artifacts/<job-id>/meta.json`

## Fallback Requirements

If API path fails and fallback is enabled:

- Node.js available
- Playwright installed in runtime environment (`npm i -D playwright` or equivalent)

Fallback script:

- `tools/paperreview_fallback.mjs`

## Notes

- Polling starts from 10 minutes by default to reduce load.
- 429 and server-pressure paths use cooldown logic.
- Jobs are persisted in SQLite at `<state_dir>/reviewloop.db`.
